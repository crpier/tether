# Health telemetry lands in a separate append-only `telemetry.sqlite3` store

The Health Connect ingestion gate (mapped in #183, resolved by this ADR under #190) is the first Telemetry source to actually land data under ADR 0012's vertical-table rule. That rule says *where in the architecture* Telemetry lives (outside the Memory pool); it does not say which database file, what the tables look like, or how updates and deletes are handled — and Health Connect's shape (many metrics, each with its own nested record/sample structure, high write volume, values that get superseded but never truly summarized away) forces those decisions now.

We land Health Connect data in a second SQLite file, `telemetry.sqlite3`, wired through `server.py` as a second snekql `Database` instance alongside the main `tether.sqlite3`. Within it, each Health Connect metric gets its own typed, `hc_`-prefixed table pair when the payload is nested — a parent table for the record and a child table for repeated sub-values, e.g. `hc_heart_rate_record` + `hc_heart_rate_sample`, `hc_sleep_session` + `hc_sleep_stage`. Rows are raw and lossless: nothing is aggregated on ingest. Aggregation is read-time work for a future Distillation layer (ADR 0012), not something the ingestion gate decides once and bakes in.

Every record row carries an `origin_id` pointing at an `hc_origin` dimension table (the writing app's package name plus device manufacturer/model/type), the Health Connect `record_uid` for the reading itself, the upstream `modified_at` timestamp, and a server-clock `received_at` timestamp. Storage is append-only: a changed or deleted upstream reading is never mutated or hard-deleted in place, it is inserted as a new version or a tombstone row. Per-table `_current` SQL views — the only in-schema read helper — expose current state by picking the latest non-tombstoned row per `record_uid`; nothing else queries the raw tables' history directly. Instant columns are stored as indexed epoch-millis UTC alongside a preserved zone-offset column, so a value's local time (which day, which side of a DST change, which timezone the phone was in) stays reconstructable rather than collapsing into ambiguous UTC.

## Considered options

- **Store Health Connect data in the main `tether.sqlite3`** — rejected: sensor-volume tables churn far more than Memory rows, which bloats backups and vacuum/checkpoint cost for the whole database over something that isn't a Memory at all. Separation is nearly free at Tether's single-tenant, single-process scale, and any future need to join telemetry against memories is an app-code (or `ATTACH DATABASE`) concern, not a reason to force them into one connection's schema.
- **A dedicated time-series database** — rejected: real ops burden (another service, another backup/restore story) for a workload this is nowhere near the scale of — single-tenant, local, one phone's worth of readings.
- **A generic EAV/key-value readings table** — rejected on two grounds: it's lossy for Health Connect's genuinely nested shapes (a sleep session's stages, a workout's samples), and ADR 0012 already commits Telemetry to typed, source-shaped vertical tables — a generic table would be an end-run around that.
- **Upsert-in-place on `record_uid`** — rejected: mutating or deleting a row destroys the previous value permanently, and Telemetry can't be backfilled from upstream once that's gone (same non-reversibility ADR 0012 already flags for ingestion schemas generally). Append-only with versions and tombstones, read through `_current` views, keeps every value Tether ever saw while still giving callers a simple "what's true now" query.

## Consequences

- `server.py` wires up two `Database` instances — main and telemetry — instead of one; anything that touches Health Connect data needs the right handle.
- `telemetry.sqlite3` is one more file in scope for encrypted off-box backups (#61).
- Charts and any future Distillation-layer reads go through the per-table `_current` views, never the raw `hc_*` tables directly, so append-only history doesn't leak into code that only wants current state.
- This ADR does not generalize Telemetry into a shared store: "typed, append-only, per-source vertical tables" remains the general pattern (ADR 0012, CONTEXT.md), and `hc_`-prefixed tables in `telemetry.sqlite3` are the Health Connect ingestion gate's specific instance of it. A future Telemetry source gets its own tables under this same pattern, not a slot in this one.

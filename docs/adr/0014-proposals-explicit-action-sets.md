# Proposals are explicitly composed action sets, host-executed; autonomy is granted explicitly per category

Proposals gate agent-initiated *action sets* — batches of consequential, reversible-in-intent operations (archive these emails, label those) produced by unattended or multi-step agent work. The design question was how a proposal comes into being, who carries it out, and how (if ever) the human's trust in a category of action lets future instances of it skip the queue.

A Proposal is **explicitly composed**, not intercepted. The agent calls a new `propose` tool with a consumer name, a title, a human-readable summary, and an explicit list of typed actions; this is the only way a gated action gets created, and it replaces the prose-enforced "Proposal-lite" convention for any category that graduates to a real gate — gated actions are simply not directly callable tools. The proposal is stored first-class in the database. On approval, **the host executes it** via per-action-kind executors; no agent is in the loop at execution time, so execution is decoupled from the agent session that proposed it and can outlive it.

Each action carries a `kind` (e.g. `gmail.archive`) and an optional consumer-defined `scope` (e.g. `sender-category:newsletter`). **Autonomy grants** are keyed on `(kind, scope)`, with a bare-kind grant covering all scopes for that kind. Matching is fail-closed: an unrecognized or typo'd scope simply doesn't match any grant, so the action queues rather than executes — scope strings are otherwise unvalidated. A proposal executes immediately only if every one of its actions is covered by an existing grant; if any action lacks a matching grant, the whole proposal queues for human review — the system never silently splits a batch into an auto-run part and a queued part.

Granting is **explicit-only**: the host computes per-`(kind, scope)` calibration statistics from proposal history (seen/approved/rejected/edited counts, most recent rejection) and surfaces them as grant *suggestions* in the UI, but only a human click creates a grant. Revocation is one click and effective immediately — grant state is read fresh, never cached, so a revocation applies to the very next proposal. Rejecting a proposal in an already-granted category prompts the human to consider revoking, but never forces it. This keeps every trust promotion in Tether a human act, consistent with ADR 0001's "AI-assisted but never AI-decided."

Lifecycle: `pending → approved | rejected`, then `approved → executing → executed | failed`. Approval can be partial — the human deselects individual actions before approving — and each action's disposition (approved/rejected/edited) is recorded and feeds the calibration stats above. Per-action outcomes (`succeeded | failed | skipped`, with detail) are appended as they happen; a failed action stays failed and visible, and re-attempting it means creating a fresh proposal, not mutating the old one. Staleness fails soft: if an action's target is already gone or already in the desired state, it resolves `skipped`, not error. There is no auto-expiry — old pending proposals just sit there. The `executing` state may be long-lived and executors may run asynchronously, which is deliberate: it lets a future class of outward actuators (e.g. authenticated-browser actions) fit the same lifecycle without redesign, and callers must never assume `approved → executed` happens as one transaction. Rejection is terminal, with an optional free-text reason.

Storage lives in the main `tether.sqlite3` (this is low-volume workflow state that wants to join with runs and notifications, not a telemetry-shaped table). Three tables: `proposal` (id, consumer, title, summary, producing-run provenance, state, timestamps, rejection reason), `proposal_action` (proposal FK, sequence, kind, scope, params as JSON, disposition, outcome, outcome detail, executed_at), and `autonomy_grant` (kind, nullable scope, granted_at, revoked_at — rows are stamped, never deleted, so a re-grant is a new row and the grant ledger doubles as trust history). Action params are **typed at the seam, JSON at rest**: each kind registers a Pydantic params model, validated once when the proposal is composed and again when the action executes. Calibration stats are computed at read time and never stored, in the spirit of ADR 0006 (search recomputed, not cached).

On the surface, a Proposals panel in the SPA mirrors the existing Review queue pattern: skimmable pending queue, expand-to-inspect, deselect individual actions, approve/reject with optimistic concurrency, plus a grants tab showing active grants, their evidence stats, a revoke control, and pending suggestions. Proposal creation notifies through the existing EventNotifier/notification path. The only agent-facing tools are `propose` and `list_proposals`; approve, reject, and grant are human-only HTTP routes, never tools, so the closed tool world (ADR 0005) makes the gate impossible to talk an agent around.

Each consumer registers, per action kind: the kind name, its Pydantic params model, an idempotent async executor, and a UI rendering hint. Idempotency matters because `executing` can crash mid-batch; a safe re-run of the remaining approved actions, with already-done ones resolving `skipped`, is what makes that safe. Registration follows the existing `*_ACTION_SPECS`-concatenated-centrally pattern, the same shape as the ToolSpec registry. The first consumer is Gmail hygiene (#199).

The gate follows authorship, the same principle used for Candidates: Proposals gate agent-initiated action sets from unattended runs and batches. Human-initiated in-chat actions already carry the human's explicit intent in the moment, so existing Proposal-lite surfaces (facet curation, panel CRUD) are left as-is for now. Whether those should eventually migrate behind the real Proposal gate, or deliberately stay on chat-approval, is deferred until after the Gmail purge proving ground.

## Considered options

- **Intercept tool calls at the ToolSpec seam** — rejected: a batch of related actions would become N separately parked calls instead of one inspectable, approvable set; execution would tie to the (possibly dead) agent session that made the calls; and it doesn't generalize to future actuators whose actions aren't instant (e.g. an authenticated-browser action that takes real time).
- **Auto-grant autonomy after N clean approvals** — rejected: this would be the only self-promoting trust gate in the system, contradicting the human-act principle everywhere else. Grants stay explicit-only until real usage data says the calibration suggestions are trustworthy enough to reconsider.
- **Per-kind typed action tables** — rejected: one table per action kind means schema explosion for what is fundamentally workflow state, not domain data. Typed-at-the-seam validation with JSON-at-rest storage gets the type safety without the table sprawl.
- **Approval via an agent-facing chat tool** — rejected: an agent could in principle be talked into calling an "approve" tool on its own proposal. Keeping approve/reject/grant off the tool surface entirely, as human-only HTTP routes, is what makes the closed tool world (ADR 0005) actually hold the gate.

## Consequences

- Every consumer must register its action kinds' params models, executors, and UI hints via `*_ACTION_SPECS` before it can call `propose` — there is no generic untyped action shortcut, a deliberate floor in the same spirit as ADR 0012's per-source vertical tables.
- Executors must be written idempotent from the start, since `executing` can be long-lived, asynchronous, and interrupted; this is a real constraint on every future consumer, not just Gmail.
- Existing Proposal-lite surfaces (facet curation, panel CRUD) remain on their current prose-enforced convention until someone deliberately migrates them; they are not automatically subsumed by this gate.
- The SPA needs a new Proposals panel plus a grants tab; this is new frontend surface, not a reuse of an existing panel.
- The `autonomy_grant` table is append-only by design (revoked_at stamps, no deletes), so it functions as a permanent trust-history log alongside its role as the live grant lookup.

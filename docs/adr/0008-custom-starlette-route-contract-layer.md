# Use a custom Starlette route-contract layer for Pydantic/OpenAPI REST

Tether exposes its host API with Starlette, but uses a small custom **route contract** layer to bind typed handler signatures, Pydantic validation/serialization, OpenAPI 3.1 generation, and Swagger UI into one source of truth. We chose this over FastAPI, SpecTree, and apispec because Tether wants Starlette's lighter transport surface while still generating API schemas from Pydantic models for the Solid client and pi tool shims.

The layer is deliberately Tether-specific rather than a reusable framework: decorated async handlers remain plain typed Python callables, request parameters are marked with `Annotated` aliases (`PathParam`, `QueryParam`, `BodyParam`), response schemas come from return annotations, status codes and expected error statuses are explicit, and public plus internal tool endpoints share one combined OpenAPI document. API contract models subclass a shared `ApiModel` so the REST contract stays separate from persistence/service models.

Handlers are attached to an **`ApiRouter`** by decorating them with the router instance: `@router("GET", "/{id}", status=...)` records the route metadata and registers the handler while returning the original callable, so no handler is ever wrapped or tied to a Starlette `Route`. An `ApiRouter` groups routes that share a resource prefix, tags, security scheme, auth error statuses, and request-context construction. An **`ApiMount`** maps runtime surface prefixes to the routers under them — `routes={"/api": [memory_router], "/internal": [tool_router]}` — and its `build_routes(title=..., version=...)` traverses every registered handler to produce the Starlette route list, the single combined OpenAPI 3.1 document, and the `/openapi.json` and `/docs` routes. The caller assembles the final app themselves: `app = Starlette(routes=mount.build_routes(...))`.

Terminology used by this decision: a **route contract** is the method, path, status, typed request parameters/body, response annotation, and expected error statuses for one handler; an **`ApiRouter`** is a set of route contracts sharing a resource prefix, tags, security, auth errors, and request-context construction; an **`ApiMount`** maps runtime surface prefixes such as `/api` or `/internal` to the routers served under them. `security` (`human_session`, `tool_secret`, or none) and `tags` are documentation/codegen metadata only — actual authentication lives in each router's `ctx_factory`. `ApiRouter` is generic over its context type: the type is inferred from `ctx_factory`'s return annotation and bound to the first parameter of every handler the router decorates, so a handler that declares the wrong context type is a static type error rather than a silent runtime mismatch.

This is hard to reverse because every REST endpoint, generated client, and generated tool shim will depend on the contract format and OpenAPI output. The rejected alternatives either brought framework behavior Tether does not want (FastAPI), hid too much behind dynamic request context (SpecTree), or still required building most of this glue to avoid runtime/spec drift (apispec).

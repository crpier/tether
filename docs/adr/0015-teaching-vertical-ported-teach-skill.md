# The teaching vertical is the ported /teach skill on typed tables, with lessons as Artifacts

The teaching vertical needed a design for curricula, lessons, and regeneration. The existing `/teach` skill already carries a proven pedagogy — Mission-grounded lessons, zone of proximal development computed from learning records, a fluency-vs-storage-strength split, and short, beautiful lessons with tight in-lesson feedback loops — built on top of workspace markdown files. The design question was how much of that pedagogy to keep and what substrate to run it on inside Tether.

The answer keeps the pedagogy unchanged and swaps only the substrate: workspace markdown files become **typed vertical tables** in the main `tether.sqlite3`, one per workspace file type: `curriculum` (topic, the MISSION fields — why / success-looks-like / constraints / out-of-scope — a teaching-preferences notes column, and state `active | completed | abandoned`, one mission per curriculum), `lesson` (curriculum FK, seq, title, `artifact_id`, `generated_at`, and regeneration lineage via a `supersedes` FK plus the weak spot it targeted — quiz results need no table of their own, since they are artifact events), `learning_record` (curriculum FK, seq, title, body, `status` of `active | superseded` plus `superseded_by` — append/supersede, never delete), `resource` (curriculum FK, kind `knowledge | wisdom | gap`, url, annotation, `pruned_at`), `glossary_term` (curriculum FK, term, definition, avoid-aliases — the one table revised in place), `reference_doc` (curriculum FK, title, `artifact_id`), plus an assets store holding the shared stylesheet and quiz components as source.

Lessons are integrated with Tether's existing Artifact, event, and run machinery rather than reinventing rendering or feedback channels. **Canonical lesson and asset source lives in the tables; Artifacts are render targets, never the source of truth.** This resolves the one impedance mismatch in the port: the `/teach` skill assumes the agent can re-read `./assets/` and prior lessons from the filesystem, but Tether agents can never read an Artifact back (ADR 0011's constraint). Generators therefore read source material from the tables and render forward into a sandboxed Artifact — never the other direction.

Lessons render as versioned sandboxed Artifacts; quizzes grade inside the Artifact's own JS for instant feedback, and per-question outcomes arrive as artifact events — the sole talk-back channel, matching the artifact-events pattern established elsewhere in the roadmap.

**Regeneration is event-driven or on-demand, never time-scheduled.** Quiz completion queues a debounced regeneration run, capped at one lesson ahead per curriculum so there is never a pile-up — this follows the no-guilt principle in `docs/principles.md`. A "next lesson" button, or an equivalent chat request, triggers regeneration manually. The run itself is an ephemeral pi run under a new **`teaching` run kind**, with a system prompt derived from the `/teach` skill and a dedicated tool belt for writing lesson rows plus their Artifacts, learning records, glossary updates, and resources. It consumes quiz events and learning records to target weak spots; it does not grade. There is no Proposal gate on this run: it is human-initiated in spirit and scoped entirely to the vertical's own tables, consistent with the "the gate follows authorship" principle from ADR 0014.

The panel (`TeachingPanel`, following the standard SolidJS panel pattern) always shows the mission, lists the lesson sequence with open-Artifact buttons and quiz-outcome summaries, offers a next-lesson button with in-flight state, and has collapsible read-only sections for learning records, glossary, and resources (editing those happens in chat, not the panel). **Progress display is positional, never temporal** — no dates-since, no counts-per-week, nothing that accrues guilt. Curriculum state transitions (active/completed/abandoned) are human clicks, not computed.

Separation from Recall is sharpened relative to the original ticket's framing. The ticket's "shared quiz UI only" is deliberately reduced to shared quiz *conventions* — same-length answer options, retrieval-first design, living in the assets store — rather than a literal shared component, because lesson quizzes are in-Artifact JS while Recall prompts are SPA components; there is no code to share between the two. Lesson quizzes remain feedback instruments; Recall remains the trust instrument.

A **one-way graduation seam into Recall is left as room, not built now**: a human-initiated "drill this into memory" action over compressed durable material (reference docs, glossary clusters) would run the existing Recall entry path (distill → loose Memory → StudyItem + SM-2 prompts). It is one-way only — Recall outcomes never flow back into curriculum progress. The sole prerequisite when this is built is generalizing `StudyItem`'s currently YouTube-coupled source field to `(source_kind, source_id)`.

## Considered options

- **Workspace files as faceted Commons Memories** — rejected: curriculum needs lifecycle state, typed queries (by curriculum, by sequence, by status), and a dedicated panel — exactly the admission criteria for a Vertical, not a Commons domain.
- **Time-scheduled regeneration cron** — rejected: a scheduled cadence would manufacture a lesson backlog whenever the human falls behind, violating the no-guilt principle. Event-driven regeneration with a one-ahead cap was chosen instead.
- **A literal shared quiz component with Recall** — rejected: lesson quizzes are in-Artifact JS and Recall prompts are SPA components, two different runtimes with no natural shared implementation. The verticals share conventions (format, retrieval-first design), not code.
- **Auto-completing lesson units from quiz scores** — rejected: quiz results are feedback instruments for the learner and the regeneration run, not gates on curriculum state. Progress changes stay human acts, consistent with ADR 0001 and ADR 0014.

## Consequences

- A new `teaching` run kind, system prompt, and dedicated tool belt must be built before any curriculum can generate lessons.
- Lesson and asset source material is stored host-side in tables rather than read back from Artifacts, so generation never depends on the agent re-reading an Artifact (which ADR 0011 forbids).
- Regeneration debounce/in-flight state must be tracked per curriculum to enforce the one-lesson-ahead cap.
- Recall graduation, when it is eventually built, requires generalizing `StudyItem`'s source key from YouTube-coupled to `(source_kind, source_id)` before it can accept curriculum-derived material.
